# OPERATION_RULES.md — 감사노트 프로젝트 운영 규칙

> 모든 기능 개발, 수정, 정책 변경 시 반드시 이 규칙을 따릅니다.  
> **문서 기록 검토가 끝나지 않으면 작업 완료로 간주하지 않습니다.**

---

## 1. 개발 전 — 필수 확인

새 작업 시작 전 반드시 아래 순서로 확인합니다.

1. `docs/PROJECT_OVERVIEW.md` — 현재 프로젝트 상태 파악
2. `docs/감사노트 프로젝트 운영체계.md` — 핵심 원칙·제약사항 확인
3. `docs/DECISIONS.md` — 기존 의사결정과의 충돌 여부 확인

**충돌 발생 시**: PM(CozyBuilder) 승인을 먼저 받습니다. 임의로 기존 결정을 번복하지 않습니다.

---

## 2. 개발 중 — 원칙

- 기존 결정사항(`DECISIONS.md`)과 충돌하는 구현은 즉시 중단하고 PM에게 보고합니다.
- 절대 변경 금지 항목(`감사노트 프로젝트 운영체계.md` §7)은 어떤 이유로도 수정하지 않습니다.

---

## 3. 개발 완료 후 — 문서 갱신 검토

기능 완료 후 반드시 아래 4개 문서의 갱신 필요 여부를 검토합니다.

### A. CHANGELOG.md 갱신 여부

아래 중 하나라도 해당되면 갱신합니다.

- [ ] 사용자가 체감할 수 있는 기능 변경
- [ ] 버전(versionCode / versionName / 앱 버전) 변경
- [ ] UI / 화면 구성 변경
- [ ] 버그 수정

### B. DECISIONS.md 기록 여부

아래 중 하나라도 해당되면 결정 항목을 추가합니다.

- [ ] 정책 결정 (기본값, ON/OFF, 임계값 등)
- [ ] 구조 변경 (파일 구조, 컴포넌트 구조)
- [ ] 아키텍처 변경 (저장 방식, 통신 방식)
- [ ] AI 사용 정책 변경
- [ ] 저장 구조 변경 (localStorage 키, 타입 변경)
- [ ] 배포 정책 변경 (Vercel, Play Store)

### C. PROJECT_OVERVIEW.md 갱신 여부

아래 중 하나라도 해당되면 갱신합니다.

- [ ] 현재 버전 변경
- [ ] 배포 주소 변경
- [ ] 프로젝트 전체 상태 변경 (빌드 상태, Play Store 상태 등)
- [ ] 다음 작업 우선순위 변경

### D. RELEASE_CHECKLIST.md 갱신 여부

아래 중 하나라도 해당되면 갱신합니다.

- [ ] 출시 절차 변경
- [ ] 배포 절차 변경 (Vercel 설정, Android 빌드 설정 등)

---

## 4. 문서 기록 원칙

> **중요한 결정은 코드보다 먼저 문서에 남긴다.**  
> **"왜 그렇게 만들었는지"가 설명되지 않으면 개발 완료로 간주하지 않는다.**

- 코드로 무엇을 했는지는 커밋으로 확인할 수 있습니다.
- 왜 그렇게 결정했는지는 `DECISIONS.md`에만 남습니다.
- 문서가 없는 결정은 다음 세션에서 번복될 위험이 있습니다.

---

## 5. 새 세션 시작 규칙

새 세션 시작 시 반드시 아래 순서로 확인 후 개발을 시작합니다.

```
1. docs/PROJECT_OVERVIEW.md          ← 현재 상태 파악
2. docs/감사노트 프로젝트 운영체계.md  ← 핵심 원칙·제약사항
3. docs/DECISIONS.md                 ← 기존 의사결정 맥락
4. docs/CHANGELOG.md                 ← 최신 버전 이력
```

확인 전에는 코드 수정을 시작하지 않습니다.

---

## 6. PM 자동 체크 — 기능 완료 보고 형식

기능 완료 보고 시 반드시 아래 항목을 함께 보고합니다.

```
## 기능 완료 보고

### 수정 파일 목록
- src/...
- ...

### 문서 반영 여부
- CHANGELOG.md 반영: Yes / No (사유: )
- DECISIONS.md 반영: Yes / No (사유: )
- PROJECT_OVERVIEW.md 반영: Yes / No (사유: )
- RELEASE_CHECKLIST.md 반영: Yes / No (사유: )

### Git
- 커밋: Yes / No (해시: )
- push: Yes / No
```

하나라도 누락 시 작업 완료로 간주하지 않습니다.

---

## 7. 규칙 적용 예외 조건

아래 작업은 문서 갱신이 불필요합니다.

- 오탈자 수정 (코드 동작 변경 없음)
- 주석 추가/수정
- 코드 포맷팅 (prettier, linting)
- 테스트용 데이터 제거 (동작 변경 없음)
- 문서 파일 자체의 수정 (CHANGELOG, DECISIONS 등)
